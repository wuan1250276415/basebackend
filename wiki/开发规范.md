# 开发规范

---

## 代码风格

### Java 代码规范

| 规则 | 说明 |
|------|------|
| Java 版本 | Java 17 语法特性（`var`、`record`、`sealed`、`switch` 表达式等） |
| Lombok | 使用 `@Data`、`@Builder`、`@Slf4j`、`@RequiredArgsConstructor` 等注解减少模板代码 |
| 代码检查 | Checkstyle + SpotBugs + SonarCloud |
| 编码格式 | UTF-8，4 空格缩进 |

### TypeScript 代码规范（前端）

| 规则 | 说明 |
|------|------|
| 严格模式 | `"strict": true` |
| 目录结构 | 按功能分组：`api/`、`pages/`、`stores/`、`components/`、`hooks/`、`utils/` |
| 状态管理 | Zustand（`stores/`） |
| 全局上下文 | React Context（`contexts/`） |
| 国际化 | i18next（`i18n/locales/`） |

## 命名规范

### Java 命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 | UpperCamelCase | `UserService`, `OrderController` |
| 方法名 | lowerCamelCase | `getUserById()`, `createOrder()` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `DEFAULT_PAGE_SIZE` |
| 包名 | 全小写 | `com.basebackend.user.service` |
| 数据库表 | lower_snake_case | `sys_user`, `sys_role` |
| 数据库字段 | lower_snake_case | `create_time`, `dept_id` |

### 分层命名约定

| 层 | 后缀 | 示例 |
|---|------|------|
| 控制器 | `Controller` | `UserController` |
| 服务接口 | `Service` | `UserService` |
| 服务实现 | `ServiceImpl` | `UserServiceImpl` |
| 数据访问 | `Mapper` | `UserMapper` |
| 实体 | 无后缀 / `Entity` | `User`, `SysUser` |
| DTO | `DTO` | `UserCreateDTO`, `UserUpdateDTO` |
| VO | `VO` | `UserVO`, `UserDetailVO` |
| 查询对象 | `Query` | `UserQuery`, `PageQuery` |

## 分层架构规范

```
Controller  →  接收请求、参数校验、调用 Service、返回 Result<T>
     │
Service(Interface)  →  定义业务接口
     │
ServiceImpl  →  实现业务逻辑、事务管理
     │
Mapper  →  数据访问（MyBatis Plus）
     │
Entity  →  数据库实体映射
```

### 各层职责

| 层 | 允许 | 禁止 |
|---|------|------|
| Controller | 参数校验（`@Valid`）、调用 Service、格式转换 | 包含业务逻辑、直接操作 Mapper |
| Service | 业务逻辑、事务管理（`@Transactional`）、调用 Mapper | 直接操作 HttpServletRequest/Response |
| Mapper | 数据库 CRUD、复杂 SQL | 包含业务逻辑 |

## 异常处理规范

### 统一异常体系

```java
// 业务异常 - 已知的业务错误
throw new BusinessException(ErrorCode.USER_NOT_FOUND);
throw new BusinessException("订单已过期，无法支付");

// 参数校验异常 - 由 @Valid 自动触发
// 全局异常处理器自动捕获并返回 400

// 系统异常 - 未知错误
// 全局异常处理器捕获并返回 500，日志记录完整堆栈
```

### 全局异常处理器

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        return Result.failure(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return Result.failure(400, message);
    }

    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        log.error("系统异常", e);
        return Result.failure(500, "系统异常，请联系管理员");
    }
}
```

### 规范要求

- 不允许在 Controller 中 `try-catch` 并吞掉异常
- Service 层抛出 `BusinessException`，由全局处理器统一处理
- 不允许返回 `null`，使用 `Optional` 或抛出异常
- 日志记录：ERROR 级别必须包含完整异常堆栈

## API 设计规范

### RESTful 规范

| HTTP 方法 | 用途 | 示例 |
|-----------|------|------|
| GET | 查询 | `GET /api/user/users/{id}` |
| POST | 创建 | `POST /api/user/users` |
| PUT | 全量更新 | `PUT /api/user/users/{id}` |
| PATCH | 部分更新 | `PATCH /api/user/users/{id}/status` |
| DELETE | 删除 | `DELETE /api/user/users/{id}` |

### 统一响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": 1708675200000
}
```

### 分页响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "records": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 10
  }
}
```

### API 文档

- 使用 Knife4j / SpringDoc OpenAPI 注解
- 每个接口必须有 `@Operation` 描述
- 参数必须有 `@Parameter` 说明

```java
@Operation(summary = "获取用户详情", description = "根据用户ID查询用户详细信息")
@GetMapping("/{id}")
public Result<UserVO> getUser(
    @Parameter(description = "用户ID", required = true) @PathVariable Long id) {
    return Result.success(userService.getById(id));
}
```

## Git 提交规范

### Commit Message 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 重构（不是新功能也不是修复） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖变更 |
| `ci` | CI/CD 配置 |
| `merge` | 合并分支 |

### 示例

```
feat(user-api): add user preference management

- Add PreferenceController with CRUD endpoints
- Add preference service and mapper
- Add Flyway migration V2.3__create_preference_table.sql

Closes #42
```

## 数据库规范

### 建表规范

- 表名：`sys_` 前缀表示系统表，业务表用业务前缀
- 主键：BIGINT 类型，雪花算法生成（ASSIGN_ID）
- 必备字段：`id`, `create_by`, `create_time`, `update_by`, `update_time`, `deleted`
- 字符集：`utf8mb4`
- 索引命名：`idx_` 前缀，唯一索引 `uk_` 前缀

### Flyway 迁移规范

- 命名：`V{major}.{minor}__description.sql`
- 每个迁移脚本幂等（使用 `IF NOT EXISTS`）
- 禁止修改已执行的迁移脚本
- 大表变更需评估锁表影响
