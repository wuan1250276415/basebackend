## 分层架构说明

### 接入层（Gateway）

- **Spring Cloud Gateway** 作为统一入口
- 过滤器链按顺序处理请求：认证 → 安全头 → 签名 → 限流 → 幂等 → 版本 → 慢请求检测 → 日志 → 缓存 → 链路追踪
- 基于 Nacos 动态路由发现
- Resilience4j 熔断器保护后端服务

### 业务层（微服务）

- 每个微服务独立部署、独立数据库
- 遵循 **Controller → Service → Mapper → Entity** 分层
- 服务间通过 **Feign** 进行 RPC 调用
- 通过 **RocketMQ** 进行异步消息通信

### 数据层

- **MySQL 8** 主数据库，Druid 连接池
- **MyBatis Plus** ORM，支持逻辑删除、自动填充
- **Flyway** 管理数据库版本迁移
- **Redis** 多级缓存（L1 Caffeine + L2 Redis）

### 基础设施层

- **Nacos**：服务注册与配置中心
- **Seata**：分布式事务（AT 模式）
- **RocketMQ**：消息队列
- **MinIO**：对象存储
- **OpenTelemetry + Prometheus + Loki + Tempo**：可观测性

## 请求流转过程

以用户登录为例，完整的请求流转过程如下：

```
1. 前端发起 POST /api/user/auth/login 请求
       │
2. Gateway 接收请求
       │
3. AuthenticationFilter 检查白名单
   ├── /api/user/auth/** 在白名单中，跳过认证
       │
4. HeaderSanitizationFilter 清理危险请求头
       │
5. SecurityHeadersFilter 添加安全响应头
       │
6. TraceIdFilter 注入链路追踪 ID
       │
7. AccessLogFilter 记录访问日志
       │
8. 路由匹配：Path=/api/user/** → lb://basebackend-user-api
       │
9. User API 的 AuthController.login() 处理
       │
10. Service 层验证用户名密码
       │
11. JWT 模块生成 Access Token + Refresh Token
       │
12. 返回 Token 给前端
```

### 已认证请求流转

```
1. 前端携带 Authorization: Bearer <token> 发起请求
       │
2. Gateway 的 AuthenticationFilter 验证 Token
   ├── 检查 Token 是否在黑名单中
   ├── 验证 Token 签名和有效期
   ├── 提取用户信息写入请求头
       │
3. 后续过滤器依次处理
       │
4. 目标微服务接收请求
   ├── JwtAuthenticationFilter 从请求头提取用户上下文
   ├── @RequiresPermission / @RequiresRole 权限校验
   ├── @DataScope 数据权限过滤
       │
5. 业务逻辑处理并返回结果
```

## 服务间通信方式

| 方式 | 技术 | 适用场景 |
|------|------|---------|
| 同步 RPC | OpenFeign | 需要实时返回结果的服务调用 |
| 异步消息 | RocketMQ | 解耦、削峰、事件通知 |
| 事务消息 | RocketMQ 事务消息 | 需要最终一致性的跨服务操作 |
| 分布式事务 | Seata AT | 需要强一致性的跨服务操作 |
| 事件驱动 | DomainEvent | 领域内事件发布与订阅 |
