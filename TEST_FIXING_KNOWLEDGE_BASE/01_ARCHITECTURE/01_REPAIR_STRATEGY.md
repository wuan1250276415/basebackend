# 修复策略框架

## 📋 概述

测试修复策略框架是基于basebackend项目实际经验总结的系统性方法论，旨在快速定位问题、选择合适的修复模式、并高效地解决问题。

## 🎯 核心原则

### 1. **分层处理原则**
```
┌─────────────────────────────────────┐
│  Controller层 (复杂)                 │  ← 跳过，避免深层依赖链
├─────────────────────────────────────┤
│  Service层 (中等)  ← 优先修复        │  ← 快速见效，容易Mock
├─────────────────────────────────────┤
│  DTO/Delegate层 (简单)               │  ← 批量处理，系统性解决
└─────────────────────────────────────┘
```

### 2. **系统性诊断原则**
1. **快速定位** - 5分钟内定位问题类型
2. **模式匹配** - 根据问题特征选择修复模式
3. **批量处理** - 相似问题统一解决
4. **验证迭代** - 逐步验证修复效果

### 3. **效率优先原则**
- 优先修复简单问题，快速提升通过率
- 复杂问题分层处理，避免陷入细节
- 建立可复用模式，减少重复工作

## 🔄 问题诊断流程

### 步骤1: 问题分类 (30秒)

根据测试日志中的错误信息，快速判断问题类型：

#### 类型A: Mock配置问题
```
错误信息特征:
- UnnecessaryStubbingException
- Missing method behavior
- NullPointerException in Mock
```
**选择模式**: 模式1(辅助方法) + 模式2(分层Mock)

#### 类型B: 查询结果为空
```
错误信息特征:
- Cannot invoke "XQuery.singleResult()" because the return value is null
- VariableInstanceQuery list() returns null
```
**选择模式**: 模式4(统一属性设置)

#### 类型C: 依赖链缺失
```
错误信息特征:
- NoSuchBeanDefinitionException
- ApplicationContext loading failed
```
**选择模式**: 模式3(Controller配置)

#### 类型D: 测试数据问题
```
错误信息特征:
- getStartTime() returns null
- Expected exception not thrown
```
**选择模式**: 模式1(辅助方法) + 手动修复

### 步骤2: 问题定位 (2分钟)

1. **查看测试类结构**
   ```bash
   # 定位测试文件
   find . -name "*Test.java" | grep -i ServiceImpl
   ```

2. **检查setUp方法**
   ```java
   @BeforeEach
   void setUp() {
       // 查看Mock配置是否完整
   }
   ```

3. **查看辅助方法**
   ```java
   private void setupXXX() {
       // 检查辅助方法是否设置完整
   }
   ```

### 步骤3: 模式选择 (1分钟)

根据问题类型和复杂程度选择最适合的修复模式：

| 问题复杂度 | 推荐模式 | 预期修复时间 |
|------------|----------|--------------|
| 简单 (Mock配置缺失) | 模式1 + 模式2 | 5-10分钟 |
| 中等 (查询结果为空) | 模式4 | 10-15分钟 |
| 复杂 (依赖链问题) | 模式3 | 30-60分钟 |
| 批量 (多个相似问题) | 模式2 (批量) | 20-30分钟 |

### 步骤4: 实施修复 (10-30分钟)

1. **应用选定模式**
2. **运行单个测试验证**
3. **批量应用类似修复**
4. **运行完整测试套件验证**
